# CLAUDE.md

本文件为 Claude Code（claude.ai/code）在此仓库中工作时提供指引。

## 项目是什么

这是一套**AI 驱动的软件产品研发协作框架**，以文档形式实现。框架定义了一套可复用的 9 步流程，由人类老板与多个专职 AI Agent 角色协作，将产品从初步想法推进到技术设计方案。所有内容以中文编写，基于 Markdown + Obsidian。本仓库不含源代码，没有构建系统，也没有测试套件。

## 流程概览

框架将 12 个步骤划分为 6 个阶段：

| 阶段 | 步骤 | 参与者 | 内容 |
|------|------|--------|------|
| 想法阶段 | 1–3 | 老板 + 产品分析师 | 老板提出想法 → Agent 扩写 → 双方对齐 |
| 需求阶段 | 4–5 | 老板 + 需求工程师 | 起草并定稿 PRD |
| 校验阶段 | 6 | 质量审核员 → 老板决策 | 校验 PRD 与完整想法是否一致 |
| 技术与设计阶段 | 7–8 | 4 个技术 Agent | Kickoff → 并行设计 |
| 设计审批阶段 | 9–10 | 质量审核员 + 老板 | 设计耦合校验 → 老板最终审批 |
| 任务拆解阶段 | 11–12 | 4 个技术 Agent + 老板 | 并行任务拆解 → 任务交叉评审 |

步骤 6 若 PRD 与完整想法存在偏离，老板可选择接受偏离或归档重启（回到步骤 1）。步骤 9 耦合校验不通过时，进入步骤 9b 全员诊断会议，按根因决定修正范围后重新校验。

## Agent 角色

| 角色 | 参与步骤 | 职责 |
|------|----------|------|
| 👔 老板 | 1,3,5,6,9b,10,12 | 人类——提供愿景、做决策 |
| 🔍 产品分析师 | 2–3 | 扩写模糊想法，提出澄清问题 |
| 📋 需求工程师 | 4–5 | 将完整想法转化为结构化 PRD |
| 🛡️ 质量审核员 | 6,9 | 校验 PRD 一致性、设计耦合校验 |
| 🏗️ 技术架构师 | 7,8a,9b,11a,12 | 系统架构设计、技术选型、任务拆解 |
| 🖥️ 前端架构师 | 7,8b,9b,11b,12 | 前端框架、页面路由、状态管理、任务拆解 |
| ⚙️ 后端架构师 | 7,8c,9b,11c,12 | API 设计、数据模型、服务架构、任务拆解 |
| 🎨 UI/UX 设计师 | 7,8d,9b,11d,12 | 设计系统、线框图、交互原型、任务拆解 |

## 目录结构

每个编号目录对应一个流程阶段：

```
00-流程与规范/          ← 流程定义与角色规范（只读参考）
01-初步想法/            ← 步骤 1：老板的初步想法
02-完整想法/            ← 步骤 2–3：产品分析师扩写（草稿 → 定稿）
03-产品需求文档/        ← 步骤 4–5：需求工程师的 PRD
04-校验记录/            ← 步骤 6：质量审核员的校验记录
05-技术与设计方案/
  ├── 技术初步共识/     ← 步骤 7：Kickoff 会议纪要
  ├── 整体架构/         ← 步骤 8a：技术架构师
  ├── 前端方案/         ← 步骤 8b：前端架构师
  ├── 后端方案/         ← 步骤 8c：后端架构师 + API 文档
  └── UI设计/           ← 步骤 8d：UI/UX 设计师
06-全局任务管理/        ← 全局任务看板与任务卡片
归档/                   ← 历史版本归档（只读）
  └── v{N}-{项目名}/
```

## 模板使用

每个文档目录下都有以 `_模板-` 开头的模板文件。在某个流程步骤中创建新文档时，复制对应模板使用，不要直接修改模板文件本身。

## 版本管理

步骤 6 校验不通过且老板选择重启时：将当前 `01`–`04` 目录的文档复制到 `归档/v{N}-{项目名}/`，然后在工作目录中重新开始。归档文档只读，不可修改。版本号 N 依次递增（如 `v1-智能待办` → `v2-...`）。
