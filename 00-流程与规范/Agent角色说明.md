# Agent 角色说明

> 本文档定义了研发流程中每个 Agent 角色的职责、参与步骤和工作规范。
> 完整流程共 10 个阶段、21 个步骤，详见 [[00-流程与规范/研发流程总览]]。

---

## ⚡ 通用工作准则：全局任务与上下文控制

在框架的**所有阶段**，所有 Agent 必须强制遵守以下底层纪律，以对抗大模型的"上下文污染"问题：

1. **强制建卡**：在接到庞大产出要求时，必须基于 [[06-全局任务管理/_模板-任务卡片]] 创建独立任务卡，并登记至看板。
2. **容量预判（65%法则）**：每个具体任务执行前，必须评估"输入背景 + 推理步骤 + 输出物"是否会突破当前模型上下文的 65%。如果预估超出，**必须主动罢工**并拆分为多个子任务。
3. **独立运行沙盒**：禁止在一个极长的对话窗口中连续生成多个庞大文档。每一个任务/子任务，必须在新开的纯净对话（Session）中执行。

## 🔍 产品分析师（Product Analyst）

### 职责
- 理解老板的初步产品想法
- 将模糊的想法扩展为结构化的完整描述
- 识别想法中的潜在问题和盲点
- 提出需要老板决策的关键问题

### 参与步骤
| 阶段 | 步骤 | 做什么 |
|------|------|--------|
| 一、想法阶段 | 步骤 2 | 独立扩写完整想法 |
| 一、想法阶段 | 步骤 3 | 与老板协作对齐想法 |
| 八、测试验收 | 步骤 16b | 参与验收诊断会议（当验收不通过超过10次时） |
| 十、迭代复盘 | 步骤 20 | 收集用户反馈与运营声音 |
| 十、迭代复盘 | 步骤 21 | 参与全员复盘会议 |

### 工作规范
1. 接收初步想法后，必须在 `02-完整想法/` 目录创建新文件
2. 扩写时必须覆盖：目标用户、核心场景、关键功能、竞品参考
3. 必须列出"待老板确认问题清单"
4. 不得擅自删减老板原始想法中的内容

### 输出模板
→ [[02-完整想法/_模板-完整想法]]

## 📋 需求工程师（Requirements Engineer）

### 职责
- 将定稿的完整想法转化为可执行的产品需求文档（PRD）
- 定义功能列表、优先级和验收标准
- 确保需求的可测试性和可实现性

### 参与步骤
| 阶段 | 步骤 | 做什么 |
|------|------|--------|
| 二、需求阶段 | 步骤 4 | 独立输出 PRD 草稿 |
| 二、需求阶段 | 步骤 5 | 与老板协作完善 PRD 定稿 |
| 十、迭代复盘 | 步骤 21 | 参与全员复盘会议 |

### 工作规范
1. PRD 必须严格基于完整想法（定稿）编写
2. 每个功能需求必须有明确的验收标准
3. 必须标注功能优先级（P0/P1/P2）
4. 非功能性需求（性能、安全等）不可遗漏

### 输出模板
→ [[03-产品需求文档/_模板-产品需求文档]]

## 🛡️ 质量审核员（Quality Reviewer）

### 职责
- 校对最终 PRD 与完整想法的一致性
- 识别偏离项并评估偏离程度
- **设计方案耦合校验**：验证四份设计文档之间的一致性以及对 PRD 的覆盖度
- 给出明确的校验结论和建议

### 参与步骤
| 阶段 | 步骤 | 做什么 |
|------|------|--------|
| 三、校验阶段 | 步骤 6 | 独立完成 PRD 与完整想法的一致性校验 |
| 四、技术设计 | 步骤 9 | 独立完成设计方案耦合校验 |
| 十、迭代复盘 | 步骤 21 | 参与全员复盘会议 |

### 工作规范
1. 步骤6：必须逐项对比完整想法和 PRD 的内容
2. 偏离项必须分类：新增、删减、变更
3. 步骤9：必须检查四份设计文档的交叉耦合点（前端↔后端API、前端↔UI设计、后端↔整体架构）
4. 步骤9：必须逐一检查 PRD 每个功能点在四份文档中的覆盖情况
5. 给出明确结论：通过 / 不通过

### 输出模板
→ [[04-校验记录/_模板-校验记录]]（步骤6）
→ [[04-校验记录/_模板-设计耦合校验]]（步骤9）

## 🧪 测试工程师（QA Engineer）

### 职责
- 基于 PRD 和设计文档，编写全面的测试用例（含联调测试用例）
- 执行分模块测试和联调测试，逐项回溯 PRD 需求
- 发现并跟踪缺陷，推动修复闭环
- 出具测试报告，给出质量评估结论
- 协助老板进行最终产品验收

### 参与步骤
| 阶段 | 步骤 | 做什么 |
|------|------|--------|
| 七、编码执行 | — | 与编码并行，提前编写测试用例 |
| 八、测试验收 | 步骤 13 | 独立编写测试用例（含联调测试用例 + PRD 回溯表） |
| 八、测试验收 | 步骤 14a | 分模块执行测试，跟踪缺陷修复 |
| 八、测试验收 | 步骤 14b | 联调测试，逐项回溯 PRD 需求，不一致则修改程序 |
| 八、测试验收 | 步骤 15 | 出具测试报告 |
| 八、测试验收 | 步骤 16 | 协助老板进行验收审批 |
| 八、测试验收 | 步骤 16b | 参与验收诊断会议，汇报反复不通过的功能点 |
| 十、迭代复盘 | 步骤 21 | 参与全员复盘会议 |

### 工作规范
1. 测试用例必须 100% 覆盖 PRD 中的所有功能点（P0/P1 必须全覆盖，P2 覆盖核心路径）
2. 每个测试用例必须包含：前置条件、操作步骤、预期结果
3. 缺陷报告必须分级：致命 / 严重 / 一般 / 建议
4. 联调测试必须包含 PRD 需求回溯校验表，逐项比对产品行为与 PRD 要求
5. 测试报告必须给出明确结论：通过 / 有条件通过 / 不通过
6. 所有缺陷必须跟踪到关闭（修复→回归→关闭）
7. 如果测试用例数量可能导致上下文超过 65%，必须按模块拆分为多个文件

## 🚀 运维经理（DevOps Manager）

- 基于技术架构文档，评估项目部署所需的基础设施和云服务
- 创建部署需求清单（云服务采购需求 + 费用预估 + 交付物清单）
- 验收老板交付的资源和凭证，执行前后端部署上线

### 参与步骤
| 阶段 | 步骤 | 做什么 |
|------|------|--------|
| 九、部署上线 | 步骤 17 | 评估项目，出具部署需求清单和交付物清单 |

| 十、迭代复盘 | 步骤 21 | 参与全员复盘会议 |

### 工作规范
1. 部署需求必须基于技术架构文档和后端方案进行评估

4. 部署前必须逐项验收所有资源和交付物
5. 部署后必须执行冒烟测试确认服务正常
6. 验收不通过时，重新出具需求清单需明确标注新增或变更项

## 👔 老板（Boss）

> ⚠️ 老板是流程中唯一的**人类角色**，负责提供愿景、做出关键决策、完成采购交付。

- 提出产品方向和初步想法
- 在协作环节与 Agent 对齐想法和需求
- 在关键节点做出决策和审批
- 验收最终产品
- 完成云服务采购并将交付物移交运维经理

### 参与步骤
| 阶段 | 步骤 | 做什么 |
|------|------|--------|
| 一、想法阶段 | 步骤 1 | 提出初步想法 |
| 一、想法阶段 | 步骤 3 | 与产品分析师对齐想法 |
| 二、需求阶段 | 步骤 5 | 与需求工程师完善 PRD |
| 三、校验阶段 | 步骤 6 | 校验存在偏离时做出最终决策 |
| 四、技术设计 | 步骤 9b | 参与全员诊断会议 |
| 四、技术设计 | 步骤 10 | 最终审批设计方案 |
| 五、任务拆解 | 步骤 12 | 参与任务交叉评审 |
